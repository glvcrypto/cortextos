## Night Mode Prep - 04:18 UTC (00:18 EDT)
### Tomorrow's upgrade proposal candidate (for 12:23 EDT cron)

**Target:** cortextOS agent boot sequence + self-verification layer

**Hypothesis:** When an agent boots, it silently assumes that all capabilities it depends on are functional (inbox polling, KB query, heartbeat write, cron restore). Today's incidents showed two cases where assumed-functional capabilities were actually not working: (1) durable cron persistence (silent failure), (2) triaged-signals state (lost across restart). A lightweight boot self-test that verifies 4-5 capability probes within the first heartbeat would surface these gaps at session start rather than mid-operation.

**Proposed implementation:** `cortextos bus self-test` — runs on session start, checks: heartbeat write, inbox poll, kb-query (1 doc), cron-list (returns >0), task-list (no error). Logs pass/fail per capability as a `session_selftest` event. If any capability fails, logs `capability_degraded` event and sends agent a notification. Takes <5 seconds.

**Expected impact:**
- Surfaces implicit-state-persistence gaps immediately on boot (would have caught triaged-signals loss)
- Surfaces durable cron failures on first restart (would have caught DEV-MAY-DURABLE-CRON earlier)
- Adds `capability_health` dimension to dashboard alongside heartbeat
- Low overhead — 5 probes at boot only, not ongoing

**Bundle candidate:** DEV-MAY-DURABLE-CRON + fleet-dedup-registry (all three are bus-layer reliability improvements)

**User review hook:** Approve / Modify (scope to specific probes) / Reject
